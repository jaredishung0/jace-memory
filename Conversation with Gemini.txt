PHASE 1: FOUNDATIONAL FILE SYSTEM TOPOGRAPHY (LOCAL LOGIC)
The baseline phase builds a human-readable, non-deterministic local workspace fully compatible with Obsidian graph visualization. It relies entirely on flat text mechanics to maintain absolute velocity and low context consumption.

1. The Core Root Maps
Five structural files live at the root of the project directory. They function as entry points into the workspace topography and are strictly hard-capped at a maximum of 50 lines / ~400 tokens to prevent context bloating.

SELF.md: Defines the primary agent's operational logic, system limits, boundaries, and active persona configuration.

HOMIE.md: Houses the human operator's profile, design methodologies, system permissions, and ongoing personal preferences.

MEMORY.md: Tracks high-level conceptual nodes, system-wide historical milestones, and overarching operational memories.

PLAN.md: Standardized strategic anchor detailing the structural "Why" and "How" of multi-step design and engineering initiatives.

TODO.md: Atomic, execution-oriented ledger utilizing markdown checkboxes (- [ ]) to track active task states.

2. Universal YAML Frontmatter Standard
Every Markdown file within the system must use structured frontmatter to permit instant scanning without consuming full file token pools.

YAML
---
id: [unique_snake_case_id]
type: [root_map | leaf_node | directory_agent | skill]
tags: [#core, #domain, #status]
synopsis: "Single-sentence high-impact functional summary."
links: [[TargetFile]], [[AlternativeFile]]
---
3. Fractal Context Mapping (Subdirectory Scoping)
The Problem: Monolithic index files expand rapidly, exceeding context targets and creating navigation noise.

The Mechanism: An AGENTS.md file is embedded directly within every subdirectory of the project tree (excluding the internal .agents/ hidden engine folder).

The Rule: These directory-level files are capped at 50 lines and contain only localized index mappings, sub-component states, and immediate leaf references using standard wiki links ([[File]]) or @file anchors.

The Scoping Result: When a runtime operates inside a specific path (e.g., /src/civil/hydrology/), it reads only the immediate local AGENTS.md, remaining completely blind to unrelated folders, saving up to 80% on active context capacity.

PHASE 2: HYBRID ENGINE & GATEWAY INTEGRATION
This layer shifts memory management from flat-file lookups to a multi-agent cognitive router powered by a hybrid bare-metal/container layout on Ubuntu 24.04 LTS.

1. Dual-Tier Runtime Environment
The Database Tier (Docker Engine): A local PostgreSQL instance runs permanently isolated inside a Docker container with restart: always. It binds exclusively to 127.0.0.1:5432 or an internal Docker bridge. It handles heavy relational persistence without cluttering the native host operating system.

The Gateway Tier (Bare-Metal uv): The LiteLLM proxy gateway runs natively on host metal, managed through the fast uv python runtime. It acts as the universal model-agnostic API router for all AI interfaces (Pi, Gemini CLI, Claude Code).

2. Standardized Memory Namespace Schema
Leveraging the underlying PostgreSQL memory architecture, all entities record context utilizing strict, clean key prefix conventions to isolate cross-agent states:

user:{user_id}:preferences: Extracted personal parameters derived from HOMIE.md.

user:{user_id}:context: Volatile operational conversation memory states.

team:{team_id}:playbook: Shared enterprise methodologies and global templates.

agent:{agent_id}:memory: Core operational histories and behavioral updates for Pi.

project:{project_id}:config: Specific hardware, architectural, or design parameters.

3. The Logical Shadow Protocol (Git-Backed Database tracking)
Because live binary database trees break traditional Git tracking, the system exports structured, deterministic, sorted textual representations to the .agents/brain/db_shadow/ tree upon task milestones.

schema.sql: Raw table schema layout tracking migration changes cleanly over time.

data_state.json: Alphabetically sorted text dumps of key-value memory data.

The Result: Git records clear text diffs of database memory states, allowing both the human operator and the primary agent to run native audits using standard git logging utilities.
